\section{Endpoints conceptuales}

\subsection{Descripcion}

Este documento define el catalogo conceptual de endpoints y operaciones esperadas del backend. Su objetivo es describir que capacidades debe exponer el API para soportar frontend y operacion administrativa, sin caer todavia en una especificacion exhaustiva tipo OpenAPI.

\subsection{Alcance}

La intencion es descriptiva. No reemplaza una especificacion exhaustiva tipo OpenAPI ni fija detalles finales de verbos HTTP, codigos de estado o esquemas fisicos.

\subsection{Definiciones}

\begin{itemize}
  \item \textbf{Endpoint conceptual}: operacion esperada del backend expresada a nivel funcional.
  \item \textbf{Operacion administrativa}: endpoint reservado a acciones autorizadas del panel de administracion.
\end{itemize}

\subsection{Reglas}

\begin{itemize}
  \item El documento describira proposito, entrada esperada y salida esperada por operacion.
  \item Los endpoints no podran contradecir reglas del dominio ni limites arquitectonicos.
  \item Toda operacion conceptual debe pertenecer claramente a un modulo responsable.
  \item El frontend consume contratos de dominio estables, no detalles internos de persistencia o integracion.
\end{itemize}

\subsubsection{Reglas de contrato conceptual}

\begin{itemize}
  \item Ninguna operacion de picks debe asumir que el frontend es autoridad de lock; el backend siempre revalida.
  \item Las operaciones de resultados deben devolver solo la verdad oficial vigente y no exponer al frontend la complejidad interna del proveedor externo.
  \item Las operaciones administrativas deben exigir autorizacion, justificacion cuando aplique y salida trazable.
  \item Las vistas de leaderboard deben responder con el ranking derivado oficial, no con datos parciales sin contexto de scoring.
\end{itemize}

\subsection{Edge Cases}

\begin{itemize}
  \item Una operacion conceptualmente valida puede terminar rechazada por estado funcional del usuario, lock o falta de permisos.
  \item Una operacion de sincronizacion o correccion de resultado puede disparar recalculo sin que el frontend invoque ese proceso de forma directa.
  \item Si un partido fue cancelado sin resultado utilizable, las operaciones de scoring derivado no deben producir puntaje asociado.
\end{itemize}

\subsection{Invariantes}

\begin{itemize}
  \item Los contratos conceptuales deben mantener consistencia entre frontend y backend.
  \item Ningun endpoint conceptual redefine reglas oficiales ya cerradas en dominio.
  \item Ninguna operacion administrativa critica existe sin posibilidad de auditoria.
\end{itemize}

\subsection{Preguntas abiertas}

\begin{itemize}
  \item Confirmar si el backend expondra operaciones separadas para predicciones especiales o si esas acciones viviran bajo un mismo grupo de picks con distincion por tipo.
\end{itemize}
