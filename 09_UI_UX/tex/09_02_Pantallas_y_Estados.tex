\section{Pantallas y estados}

\subsection{Descripcion}

Este documento define el inventario funcional de pantallas y los estados visibles minimos que cada una debe contemplar. Su objetivo es reducir ambiguedad entre flujo, contrato y comportamiento visible del frontend.

\subsection{Alcance}

Aplica a vistas publicas, autenticadas y administrativas de v1. No define wireframes finales, pero si que estados deben existir para que la experiencia sea correcta.

\subsection{Definiciones}

\begin{itemize}
  \item \textbf{Estado de pantalla}: combinacion de datos, permisos y contexto funcional que altera el comportamiento visible de una vista.
  \item \textbf{Modo solo lectura}: estado en el que la informacion es visible pero no editable por regla del sistema.
\end{itemize}

\subsection{Reglas}

\begin{itemize}
  \item Cada pantalla debe declarar explicitamente estados de carga, vacio, error y restriccion cuando apliquen.
  \item Los estados visibles deben alinearse con estados reales del backend.
  \item Un cambio de estado funcional relevante debe reflejarse sin obligar al usuario a inferirlo.
\end{itemize}

\subsubsection{Estados visibles especialmente sensibles}

\begin{itemize}
  \item \texttt{PENDIENTE}, \texttt{ACTIVO}, \texttt{RECHAZADO} y \texttt{BLOQUEADO} en activacion y contexto de cuenta.
  \item Partido editable o bloqueado en captura de picks.
  \item \texttt{SIN\_PICK} como ausencia valida de prediccion guardada.
  \item Resultado corregido cuando el sistema deba comunicar una actualizacion relevante.
  \item Accion administrativa restringida cuando falten permisos o contexto funcional valido.
\end{itemize}

\subsubsection{Reglas para estados de error}

\begin{itemize}
  \item Un error funcional debe indicar que regla impidio la accion.
  \item Un error tecnico debe permitir distinguir que la falla no fue del usuario cuando eso sea cierto.
  \item Un error no debe dejar la pantalla en un estado ambiguo sobre si la accion se aplico o no.
\end{itemize}

\subsubsection{Reglas para estados de carga y vacio}

\begin{itemize}
  \item La carga debe sugerir que dato se esta esperando cuando eso mejore comprension.
  \item Un estado vacio debe orientar al siguiente paso util y no parecer una falla silenciosa.
  \item El vacio de picks no debe confundirse con \texttt{SIN\_PICK} ya consolidado tras el lock.
\end{itemize}

\subsection{Edge Cases}

\begin{itemize}
  \item Una pantalla puede pasar de editable a solo lectura entre carga inicial y envio de accion por un lock reciente; la vista debe reconciliarse correctamente.
  \item Un leaderboard puede requerir refresco visible cuando un resultado corregido cambie posiciones.
  \item Una pantalla administrativa puede tener acceso valido al modulo pero restricciones a una accion puntual segun permiso o contexto.
\end{itemize}

\subsection{Invariantes}

\begin{itemize}
  \item El frontend no muestra un estado funcional que el backend no pueda sostener.
  \item Toda pantalla critica tiene un estado visible para exito, restriccion y error.
  \item Los estados visibles reducen ambiguedad operativa y no la aumentan.
\end{itemize}

\subsection{Preguntas abiertas}

\begin{itemize}
  \item Confirmar si el detalle de partido existira como pantalla propia o si toda la experiencia de picks se resolvera desde una unica superficie de calendario.
  \item Confirmar en que superficies exactas vivira el aviso de resultado corregido.
\end{itemize}

\subsection{Decisiones pendientes}

\begin{itemize}
  \item \texttt{Decision pendiente} \texttt{Bloquea frontend}: cerrar las superficies exactas donde se muestra el aviso de resultado corregido al participante.
  \item \texttt{Decision pendiente} \texttt{No bloqueante}: definir si el leaderboard por fase o jornada se representa como pantalla propia o como variante filtrada de una unica vista.
\end{itemize}
