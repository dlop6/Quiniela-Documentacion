\section{Roadmap y fases}

\subsection{Descripcion}

Este documento fija la planeacion operativa del proyecto hasta la salida previa al Mundial del 11 de junio de 2026. Se enfoca en la secuencia semanal de trabajo, los hitos duros, el valor visible para el usuario y los bloqueadores que no deben empujarse a finales de mayo.

\subsection{Alcance}

Aplica a la coordinacion entre producto, backend, frontend, testing y operacion para la v1 de Quiniela.

\subsection{Reglas}

\begin{itemize}
  \item El roadmap se organiza por reduccion de riesgo funcional y operativo.
  \item Los tickets pueden cambiar, pero el \textbf{goal} del sprint no debe romperse sin replaneacion explicita.
  \item Toda decision bloqueante debe cerrarse antes del sprint que la necesita.
  \item A partir del 19 de mayo de 2026 no entran features nuevas; solo correcciones y preparacion de salida.
  \item El buffer de finales de mayo es obligatorio y no debe consumirse con alcance nuevo.
\end{itemize}

\subsection{Planeacion por sprint}

\begin{docnote}
\textbf{Sprint 2}\\
\textbf{Fechas}: 7 al 13 de abril de 2026\\
\textbf{Epicas foco}: \texttt{E1 Fundations}, \texttt{E4 Payments and Activation}

\textbf{Goal}\\
Cerrar el slice funcional de activacion para que un participante pueda avanzar desde acceso hasta estado visible y validable, con control administrativo y gating basico.

\textbf{El usuario podra}\\
\begin{itemize}
  \item registrarse e iniciar sesion;
  \item recuperar acceso;
  \item ver su estado de activacion;
  \item consultar como pagar;
  \item enviar comprobante;
  \item entender si su cuenta esta \texttt{PENDIENTE}, \texttt{ACTIVO}, \texttt{RECHAZADO} o \texttt{BLOQUEADO}.
\end{itemize}

\textbf{Definition of Done}\\
\begin{itemize}
  \item existe flujo funcional de activacion de extremo a extremo;
  \item la operacion administrativa minima de aprobacion y rechazo esta soportada;
  \item el sistema bloquea comportamiento no permitido por estado;
  \item no quedan ambiguedades criticas sobre activacion que bloqueen el siguiente sprint.
\end{itemize}

\textbf{Exit criteria}\\
\begin{itemize}
  \item demo funcional del flujo de activacion;
  \item backlog del sprint siguiente actualizado;
  \item criterios de aceptacion y testing del flujo ya reflejados.
\end{itemize}
\end{docnote}

\begin{docnote}
\textbf{Sprint 3}\\
\textbf{Fechas}: 14 al 20 de abril de 2026\\
\textbf{Epicas foco}: \texttt{E2 Core Quiniela}

\textbf{Goal}\\
Dejar lista la base funcional para consultar partidos y preparar el flujo de picks.

\textbf{El usuario podra}\\
\begin{itemize}
  \item ver el calendario o listado de partidos;
  \item identificar kickoff y estado visible del partido;
  \item navegar la superficie desde la que luego se capturaran picks.
\end{itemize}

\textbf{Definition of Done}\\
\begin{itemize}
  \item el participante puede consultar partidos de forma consistente con el calendario oficial;
  \item el sistema tiene base clara para la captura de picks;
  \item no se inventan estados o agrupaciones no documentadas.
\end{itemize}

\textbf{Exit criteria}\\
\begin{itemize}
  \item demo del listado visible de partidos;
  \item contratos suficientemente estables para el trabajo de picks;
  \item backlog refinado del sprint siguiente con foco en captura y lock.
\end{itemize}
\end{docnote}

\begin{docnote}
\textbf{Sprint 4}\\
\textbf{Fechas}: 21 al 27 de abril de 2026\\
\textbf{Epicas foco}: \texttt{E2 Core Quiniela}

\textbf{Goal}\\
Cerrar la integridad operativa del flujo de picks: captura valida, edicion permitida y lock automatico confiable.

\textbf{El usuario podra}\\
\begin{itemize}
  \item crear picks por partido;
  \item editar picks antes del lock;
  \item ver cuando un partido esta editable o bloqueado;
  \item quedar en \texttt{SIN\_PICK} si no guardo antes del lock;
  \item operar en modo solo lectura cuando el pick ya no sea editable.
\end{itemize}

\textbf{Definition of Done}\\
\begin{itemize}
  \item los picks pueden capturarse y persistirse de forma consistente;
  \item el lock automatico esta definido e implementado sin ambiguedad funcional;
  \item el comportamiento de lock esta cubierto por pruebas criticas;
  \item el sistema queda listo para conectar resultados y scoring.
\end{itemize}

\textbf{Exit criteria}\\
\begin{itemize}
  \item demo de captura, edicion y cierre de picks;
  \item evidencia de pruebas para rechazo por lock y por usuario no activo;
  \item backlog refinado de resultados y scoring.
\end{itemize}
\end{docnote}

\begin{docnote}
\textbf{Sprint 5}\\
\textbf{Fechas}: 28 de abril al 4 de mayo de 2026\\
\textbf{Epicas foco}: \texttt{E5 Results Integration and Manual Fallback}

\textbf{Goal}\\
Establecer la autoridad del resultado oficial y el flujo operativo de sincronizacion, correccion manual y trazabilidad minima.

\textbf{El usuario podra}\\
\begin{itemize}
  \item ver resultados oficiales visibles de partidos;
  \item depender de una unica verdad vigente del resultado;
  \item recibir la consecuencia visible de una correccion de resultado cuando aplique.
\end{itemize}

\textbf{Definition of Done}\\
\begin{itemize}
  \item existe una ruta clara para obtener o consolidar resultados oficiales;
  \item esta resuelta la precedencia entre fuente externa y override manual;
  \item existe trazabilidad suficiente de acciones criticas sobre resultados;
  \item el proyecto ya puede alimentar scoring con una base confiable de resultados.
\end{itemize}

\textbf{Exit criteria}\\
\begin{itemize}
  \item demo de resultado oficial visible y correccion controlada;
  \item evidencia de recalculo coherente despues de correccion;
  \item backlog refinado de scoring y leaderboard.
\end{itemize}
\end{docnote}

\begin{docnote}
\textbf{Sprint 6}\\
\textbf{Fechas}: 5 al 11 de mayo de 2026\\
\textbf{Epicas foco}: \texttt{E3 Scoring and Leaderboard}

\textbf{Goal}\\
Cerrar el primer ciclo competitivo completo: picks, resultados, scoring y leaderboard visible.

\textbf{El usuario podra}\\
\begin{itemize}
  \item ver su puntaje acumulado;
  \item ver el leaderboard general;
  \item consultar leaderboard filtrado por jornada o fase;
  \item participar con picks y predicciones especiales que ya impactan el total competitivo.
\end{itemize}

\textbf{Definition of Done}\\
\begin{itemize}
  \item el sistema calcula scoring conforme a reglas documentadas;
  \item multiplicadores y desempates criticos estan implementados correctamente;
  \item el leaderboard general funciona con consistencia funcional;
  \item los casos criticos de scoring y leaderboard tienen cobertura razonable.
\end{itemize}

\textbf{Exit criteria}\\
\begin{itemize}
  \item demo de ciclo competitivo completo;
  \item evidencia de pruebas sobre scoring, lock y correccion de resultados;
  \item backlog de hardening reducido a gaps concretos.
\end{itemize}
\end{docnote}

\begin{docnote}
\textbf{Sprint 7}\\
\textbf{Fechas}: 12 al 18 de mayo de 2026\\
\textbf{Epicas foco}: \texttt{E7 QA, Accessibility, and Hardening}

\textbf{Goal}\\
Reducir riesgo real de salida: endurecer seguridad, cerrar huecos criticos y estabilizar los flujos sensibles.

\textbf{El usuario podra}\\
\begin{itemize}
  \item usar una plataforma mas estable y consistente;
  \item ver restricciones y errores mejor comunicados;
  \item confiar mas en la coherencia del sistema antes de salida real.
\end{itemize}

\textbf{Definition of Done}\\
\begin{itemize}
  \item el set minimo de seguridad de v1 esta resuelto;
  \item los flujos criticos tienen validacion suficiente segun riesgo;
  \item los huecos principales de accesibilidad y estados visibles ya no bloquean salida;
  \item el sistema esta apto para entrar en fase de release candidate.
\end{itemize}

\textbf{Exit criteria}\\
\begin{itemize}
  \item evidencia de validacion transversal en flujos criticos;
  \item lista corta y priorizada de bugs restantes;
  \item decisiones pendientes bloqueantes reducidas a cero.
\end{itemize}
\end{docnote}

\begin{docnote}
\textbf{Sprint 8}\\
\textbf{Fechas}: 19 al 25 de mayo de 2026\\
\textbf{Epicas foco}: cierre transversal de \texttt{E1}--\texttt{E7}

\textbf{Goal}\\
Entrar en \texttt{feature freeze} y dejar una version candidata a salida, con enfoque total en bugs, aceptacion y preparacion operativa.

\textbf{El usuario podra}\\
\begin{itemize}
  \item usar una version practicamente final de la plataforma;
  \item recorrer el flujo principal sin huecos criticos;
  \item operar sobre una experiencia estable de registro, activacion, picks, resultados y leaderboard.
\end{itemize}

\textbf{Definition of Done}\\
\begin{itemize}
  \item no entran features nuevas;
  \item los bugs relevantes estan resueltos o triageados explicitamente;
  \item existe validacion funcional integral de los flujos principales;
  \item el equipo puede declarar \texttt{code complete} al cierre del sprint.
\end{itemize}

\textbf{Exit criteria}\\
\begin{itemize}
  \item version candidata a salida demostrable de punta a punta;
  \item backlog separado entre bugs bloqueantes y trabajo post-salida;
  \item confirmacion explicita de \texttt{feature freeze}.
\end{itemize}
\end{docnote}

\subsection{Bloqueadores transversales}

\begin{itemize}
  \item \textbf{Bloquea backend}: cerrar el set minimo obligatorio de autenticacion, gestion de sesion y proteccion contra abuso para v1.
  \item \textbf{Bloquea frontend}: definir el flujo visual de predicciones especiales dentro del modulo de picks.
  \item \textbf{Bloquea frontend}: definir el desglose visual de puntos en leaderboard, perfil y detalle de rendimiento.
  \item \textbf{Bloquea frontend}: cerrar las superficies exactas donde se muestra el aviso de resultado corregido.
\end{itemize}

\subsection{Capacidad reservada}

\begin{itemize}
  \item Entre Sprint 2 y Sprint 3 la capacidad puede priorizar construccion.
  \item A partir de Sprint 4 se recomienda reservar entre 20\% y 30\% para bugs, integracion, retrabajo y ajustes documentales.
  \item A partir de Sprint 7 la prioridad es estabilidad, no expansion funcional.
\end{itemize}

\subsection{Invariantes}

\begin{itemize}
  \item No se mueve la fecha de \texttt{code complete} sin aceptar explicitamente un aumento de riesgo.
  \item Ningun sprint posterior a Sprint 6 debe depender de descubrimiento funcional fuerte.
  \item El buffer de finales de mayo existe para estabilizar, no para seguir construyendo.
\end{itemize}

\subsection{Decisiones pendientes}

\begin{itemize}
  \item \texttt{Decision pendiente} \texttt{No bloqueante}: decidir si este roadmap semanal se actualizara sprint a sprint como documento vivo o si solo registrara grandes cortes de avance.
\end{itemize}
